01Molten dictation

With a streaming provider, words land in the focused field while you’re still speaking. Each word arrives molten — the provider is allowed to change its mind about it. When it does, the daemon backspaces to the divergence and retypes: the text repairs itself in front of you, in the real field, as real keystrokes.

A word that survives the stability window freezes and is never touched again — repairs stay short, and your caret never runs away. max_molten_chars bounds the revisable tail outright. While you record, the GNOME overlay pill becomes a live caption: a small VU meter plus the molten tail of the transcript.

Live typing needs a streaming provider (xai, hyperfurion) — or any local server via live_rest, which re-transcribes the growing recording on an interval. REST providers still get the whole grammar below; their words just land on stop. And flow.enabled = false restores the old record → wait → type behavior, exactly.

Try it above — and try slowing stability_ms to 4000 in the live config to watch words stay molten longer. Hands-free: auto_stop_ms = 1200 ends the recording by itself after ~1.2 s of silence. Tap, speak, done.

02The spoken grammar

Edits, punctuation, and escapes are commands, not text — parsed deterministically before anything is typed, on every provider, streaming or not. Everything below works in the demo up top; say it, or press a chip.

You sayYou get
“scratch that” · “delete that” deletes the last utterance — even after it’s been typed
“new line” · “new paragraph” a line break · a blank line
“period” · “comma” · “em dash” · “open quote” · … . , — " — the glyph, correctly spaced (30-odd phrases)
“literal period” the word “period” — the escape hatch
“twenty three” 23 — in the terminal register, or with numbers = "always". Decimals too (“three point one four” → 3.14) and digit runs (“one two seven” → 127)
“furion, make that formal” rewrites what you just dictated, in place — the rewrite lane

Every phrase is remappable and removable, and your own vocabulary expands as you speak:

[flow.commands]
scratch_that = ["nuke it", "scratch that"]

[flow.punctuation]
"comma" = ""                    # fine, have it your way

[flow.vocabulary]
"hyper furion" = "HyperFurion"  # live in the demo above — try saying it

03Registers & the focus probe

Prose wants capital letters; a shell does not. At recording start the daemon probes the focused app — AT-SPI on Linux, Quartz on macOS, Win32 on Windows — and picks how dictation should render there:

RegisterBehavior
prose default smart capitalization, punctuation spacing
terminal no auto-caps, spoken numbers become digits, pastes with Ctrl+Shift+V
verbatim grammar off — words exactly as recognized

Known terminals map automatically — kitty, alacritty, foot, konsole, GNOME Terminal, wezterm, ghostty, Windows Terminal, iTerm2, and friends — and you can override or extend per app:

[registers.map]
kitty = "terminal"
code = "verbatim"

The demo’s two windows carry different registers — dictate “twenty three failed tests” into both and the register is the difference you see.

The focus guard. If focus moves to a different app mid-dictation, typing freezes immediately and the transcript lands on the clipboard instead. Dictation never types into the wrong window — toggle the demo’s windows while recording and watch it refuse.

Full Unicode on Linux

The uinput injector now types anything: plain ASCII takes the fast key path, and any other run — accents, CJK, emoji, em-dashes — is pasted through the clipboard (wl-copy on Wayland, xclip on X11) with your previous clipboard contents restored afterwards. Terminals get the Ctrl+Shift+V chord via the register. The macOS and Windows backends always typed full Unicode natively; now Linux keeps up.

04The rewrite lane

End an utterance with the wake word — or speak it alone — and what you just dictated is rewritten in place, on screen, by an LLM:

“we fixed a bunch of bugs and it works now”
“… furion, make that formal”
→ “We resolved several defects and it functions correctly.”

The instruction is anything you can say — make that formal, friendlier, shorter, a bullet list. The same channel is a command, any time:

voice-keyboard transform "make it friendlier"
voice-keyboard history 10    # the ledger (opt-in)
voice-keyboard recall 2      # re-type an entry

The rewrite goes through [llm] — any OpenAI-compatible chat endpoint. The default is xAI’s grok-4-fast; point base_url at a local llama.cpp or vLLM server and rewrites are fully local:

[llm]
provider = "xai"          # or any /chat/completions endpoint
model = "grok-4-fast"
# base_url = "http://localhost:8080/v1"   # fully-local rewrites

The just-typed text and your instruction leave for the [llm] provider only when you invoke the wake word or transform — never otherwise. The dictation ledger is off by default; opt in with history = true and it is a local file, mode 600. voice-keyboard status reports provider, register, flow state, focused app, and the last error. Edits to [flow], [registers], and [llm] hot-reload at the next recording — no daemon restart.

05What leaves your machine

If you tried the demo, you just spoke into a web page — and the section above mentioned an LLM. A fair moment to ask where everything goes. Both halves of the answer are short.

The daemon

  • Dictated audio goes to your chosen STT provider; selected text goes to your TTS provider. Nothing else, nowhere else.
  • Wake-word rewrites send that one utterance plus your instruction to your [llm] provider — only when you invoke them.
  • Recorded audio is held in memory while it is being transcribed — not written to disk.
  • TTS audio is a temporary MP3, deleted after playback.
  • The dictation ledger is opt-in, local, mode 600 — off by default.
  • API keys sit in config.toml, mode 600. The IPC socket is mode 600.
  • Default logs record state and transcript lengths, not transcript text.
  • input-group membership is broad input access — that is the honest price of global hotkeys.

This page

  • Dictation audio goes to your browser’s speech engine, the same way the daemon sends audio to its provider.
  • TTS uses your browser’s speech synthesis.
  • The wake-word rewrite here is a deterministic page stand-in — labeled in the terminal, nothing leaves.
  • No scripts loaded from anyone, no analytics, no cookies, no storage.
  • Nothing is loaded, stored, or sent — until you opt in: the terminal’s real, ask, say, and demo commands talk to api.hyperfurion.com (real xAI engines, rate-limited, labeled in place). Nothing else on this page touches the network, ever.
  • Transcripts exist only on your screen and die on reload.

06Signal path

A local daemon owns the long-lived resources: microphone capture, the speech-to-text client, the flow engine, a uinput virtual keyboard, the global hotkey listener, and an IPC socket. When you dictate, audio streams to the provider you configured; the transcript comes back token by token, passes through the grammar and the focused app’s register, and is typed into that app as real keystrokes — terminals, IDEs, browsers, anything, on Wayland and X11.

The reverse lane reads your desktop’s primary selection, sends it to a text-to-speech provider, and plays the audio locally.

With a streaming provider the transcript types while you speak; a REST provider lands it on stop. ASCII goes through the fast uinput key path; everything else — accents, CJK, emoji — pastes through the clipboard and your clipboard is restored after. The daemon is one process; the CLI is deliberately small — it reads config, sends one IPC command over a Unix socket, and exits.

09Configuration

The real config lives at ~/.config/voice-keyboard/config.toml (mode 600 — it holds your API keys). On the daemon, [flow], [registers], and [llm] hot-reload at the next recording. The copy below is live: edit it and this page re-binds itself. Slow stability_ms, set auto_stop_ms = 1200, rename the wake_word, switch the default register to verbatim — then dictate again.

[hotkey]
key = "control+alt+v"
# auto = tap to toggle, hold to talk
mode = "auto"
hold_threshold_ms = 280

[flow]
# false = the old behavior: record, stop, then type
live = true
# a molten word freezes after surviving this long
stability_ms = 1500
# 1200 = hands-free: tap, speak, done (0 = off)
auto_stop_ms = 0
# spoken cardinals → digits: auto (terminal register),
# always, or off
numbers = "auto"
# "<wake_word>, <instruction>" rewrites via [llm]
wake_word = "furion"

[registers]
# the editor window's register: prose | terminal | verbatim
# (the daemon probes the focused app; its terminal list is
# built in — the demo's terminal is always "terminal")
default = "prose"

[stt]
# the daemon also takes provider = "xai" (default)
#   | "hyperfurion" | "openai" | "groq" | "deepgram"
#   | "assemblyai" — this page's provider is your
#   browser's speech engine, so only these two apply:
language = "en"
interim_results = true

[tts]
# daemon keys: provider, model, voice_id, language;
# here voice_id matches your browser's installed voices
voice_id = "eve"
rate = 1.0
pitch = 1.0

On the desktop, provider/key/audio/hotkey changes still want a restart: systemctl --user restart voice-keyboard-daemon.

The desktop hotkey listener reads Linux input events directly — that is why your user needs input-group access, and why you log out and back in once after install. On macOS it’s an event tap; on Windows, a low-level keyboard hook — which is also why the daemon’s own typing can never re-trigger the hotkey there.

11Honest limitations

known limits — stated, not hidden

  • Linux first — Python 3.11+, systemd user services, uinput access. macOS and Windows are beta (native Quartz/SendInput backends; see the README). iOS can’t host a system-wide keyboard, and this page won’t pretend otherwise.
  • Cloud by default; no speech model ships in the box. Fully offline is one config line — point providers.openai.base_url at a local OpenAI-compatible server (see Providers above).
  • Live typing needs a streaming provider; REST providers land the text on stop. live_rest = "always" re-bills a cloud REST provider on every interim probe — that is why only local servers get it by default.
  • Live repairs assume nothing else edits the field mid-dictation: type or click into the text while speaking and a repair can land in the wrong place. The stability window and max_molten_chars bound the damage.
  • “scratch that” declines to delete across complex Unicode — emoji, combining marks — where backspace-per-character isn’t reliable.
  • A finalize that retro-revises an already-frozen word keeps the frozen form; only the still-molten tail adopts late revisions.
  • Non-ASCII on Linux types through a clipboard paste (needs wl-copy or xclip); clipboard managers may glimpse the transient entry before the old contents are restored.
  • The near-field recording overlay — and its live molten caption — is GNOME Shell 50 on Wayland; other desktops get ordinary notifications.
  • Stop-to-text latency depends on your provider; with a streaming provider the wait mostly disappears, because the text is already there.
  • IPC commands are handled one at a time; a slow TTS request can delay the next command.
  • This page runs on your browser’s speech engines, the same way the daemon runs on its providers. When the browser has no speech engine or you deny the mic, the page says so and improvises — honestly labeled. Type real in the terminal to switch the mic to the actual xAI engine via the hosted demo — opt-in, rate-limited, and labeled while it runs.

12Install

Linux

curl -L https://github.com/liamghennigan/HyperFurion-VK/releases/latest/download/install-hyperfurion-vk.sh -o install-hyperfurion-vk.sh
chmod +x install-hyperfurion-vk.sh
./install-hyperfurion-vk.sh

First run (Linux)

  1. Make sure ~/.local/bin is on your PATH: command -v voice-keyboard
  2. Log out and back in once, so input-group access takes effect.
  3. systemctl --user restart voice-keyboard-daemon
  4. voice-keyboard status — it prints idle or recording.

Upgrading is the same command — the installer restarts the daemon so the version you installed is the version that’s running.

macOS — beta

git clone https://github.com/liamghennigan/HyperFurion-VK
cd HyperFurion-VK
./packaging/macos/install-macos.sh

Native Quartz injection (full Unicode) and event-tap hotkeys, running as a launchd agent. macOS will ask for two permissions — Accessibility and Microphone — the installer walks you through both. Status arrives as notification-center toasts.

Windows — beta

git clone https://github.com/liamghennigan/HyperFurion-VK
cd HyperFurion-VK
powershell -ExecutionPolicy Bypass -File packaging\windows\install-windows.ps1

Native SendInput injection (full Unicode) and a low-level keyboard hook, started at login from your Startup folder — no admin required. IPC runs on loopback TCP guarded by a per-session token file, so other local processes can’t drive your keyboard. Beta on both platforms means exactly that: unit-tested, honest about being young.

Linux needs the uinput module, Python 3.11+, and systemd user services; the installer supports apt-get, dnf, and pacman. Full walkthroughs, non-interactive install, and troubleshooting live in the README.
