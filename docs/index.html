01Signal path

A local daemon owns the long-lived resources: microphone capture, the speech-to-text client, a uinput virtual keyboard, the global hotkey listener, and an IPC socket. When you dictate, audio goes to the provider you configured and the transcript is typed into the focused app as real keystrokes — terminals, IDEs, browsers, anything, on Wayland and X11.

The reverse lane reads your desktop’s primary selection, sends it to a text-to-speech provider, and plays the audio locally.

The typed transcript is injected as printable ASCII plus newline and tab. The daemon is one process; the CLI is deliberately small — it reads config, sends one IPC command over a Unix socket, and exits.

02What leaves your machine

If you tried the demo, you just spoke into a web page — a fair moment to ask where that went. Both halves of the answer are short.

The daemon

  • Dictated audio goes to your chosen STT provider; selected text goes to your TTS provider. Nothing else.
  • Recorded audio is held in memory while it is being transcribed — not written to disk.
  • TTS audio is a temporary MP3, deleted after playback.
  • API keys sit in config.toml, mode 600. The IPC socket is mode 600.
  • Default logs record state and transcript lengths, not transcript text.
  • input-group membership is broad input access — that is the honest price of global hotkeys.

This page

  • Dictation audio goes to your browser’s speech engine, the same way the daemon sends audio to its provider.
  • TTS uses your browser’s speech synthesis.
  • One file, no scripts loaded, no analytics, no cookies, no storage.
  • Nothing is loaded, stored, or sent — until you opt in: the terminal’s real, ask, say, and demo commands talk to api.hyperfurion.com (real xAI engines, rate-limited, labeled in place). Nothing else on this page touches the network, ever.
  • Transcripts exist only on your screen and die on reload.

04Three ways to run it

All three run the same product. Nothing on this page is feature-gated, and nothing below changes that.

Your own key — free, forever

Bring an API key from any provider above and you have every feature this product will ever ship. One line in the config. Paying unlocks nothing — that is the point.

Hosted — $5 or $10 a month

provider = "hyperfurion" buys convenience, not capability. Every feature of this product is open source and free forever — bring your own API key and you have all of it; paying unlocks nothing. What $5/month gets you is one key instead of provider accounts (20 hours of streaming dictation and 250k spoken characters a month, hard-capped so nobody can run up a bill; $10/month buys 60 h and 1M) — and what’s left over funds this project’s development. Your audio takes the same path as the default provider’s, through a metered relay to xAI; it is held in memory, never written to disk. The relay itself is open source and self-hostable. Launching soon — type subscribe in the terminal above, or get early access via sponsors.

Fully offline, if you want it

The openai provider takes a base_url, so any OpenAI-compatible server counts as a provider — including one on localhost. No API key required, nothing leaves your machine:

[providers.openai]
base_url = "http://localhost:8000/v1"  # a local server — no key needed

[stt]
provider = "openai"  # set stt.model to the model id your server exposes

The strongest open models to serve locally right now: Speaches is the easiest single server — OpenAI-compatible STT and TTS, running Whisper large-v3-turbo for dictation and Kokoro-82M for speech. NVIDIA Parakeet TDT tops the Open ASR leaderboard and is absurdly fast on a GPU; Voxtral Mini 3B (Apache-2.0) serves OpenAI-compatible via vLLM; and whisper.cpp covers CPU-only and edge boxes.

05Configuration

The real config lives at ~/.config/voice-keyboard/config.toml (mode 600 — it holds your API keys). The copy below is live: edit it and this page re-binds itself. Change the hotkey, switch the mode, set a language, pick a voice_id — then try it.

[hotkey]
key = "control+alt+v"
# auto = tap to toggle, hold to talk
# other choices: toggle, hold
mode = "auto"
hold_threshold_ms = 280

[stt]
# the daemon also takes provider = "xai" (the default)
#   | "hyperfurion" (subscription) | "openai" | "groq"
#   | "deepgram" | "assemblyai" — this page's provider is
#   your browser's speech engine, so only these two apply:
language = "en"
interim_results = true

[tts]
# the daemon also takes provider = "xai" (the default)
#   | "openai" | "elevenlabs"; here voice_id is matched
#   against your browser's installed voices instead
voice_id = "eve"
# rate and pitch are page-only knobs (the daemon's real
# keys are provider, model, voice_id, language)
rate = 1.0
pitch = 1.0

On the desktop, restart after editing: systemctl --user restart voice-keyboard-daemon.

07Honest limitations

known limits — stated, not hidden

  • Linux first — Python 3.11+, systemd user services, uinput access. macOS and Windows are beta (native Quartz/SendInput backends; see the README). iOS can’t host a system-wide keyboard, and this page won’t pretend otherwise.
  • Cloud by default; no speech model ships in the box. Fully offline is one config line — point providers.openai.base_url at a local OpenAI-compatible server (see Providers above).
  • Injection is printable ASCII plus newline and tab. Accents, CJK, emoji are skipped with warnings in the journal.
  • The near-field recording overlay is GNOME Shell 50 on Wayland; other desktops get ordinary notifications.
  • Stop-to-text latency depends on your provider and connection.
  • IPC commands are handled one at a time; a slow TTS request can delay the next command.
  • This page runs on your browser’s speech engines, the same way the daemon runs on its providers. When the browser has no speech engine or you deny the mic, the page says so and improvises — honestly labeled. Type real in the terminal to switch the mic to the actual xAI engine via the hosted demo — opt-in, rate-limited, and labeled while it runs.

08Install

Linux

curl -L https://github.com/liamghennigan/HyperFurion-VK/releases/latest/download/install-hyperfurion-vk.sh -o install-hyperfurion-vk.sh
chmod +x install-hyperfurion-vk.sh
./install-hyperfurion-vk.sh

First run (Linux)

  1. Make sure ~/.local/bin is on your PATH: command -v voice-keyboard
  2. Log out and back in once, so input-group access takes effect.
  3. systemctl --user restart voice-keyboard-daemon
  4. voice-keyboard status — it prints idle or recording.

macOS — beta

git clone https://github.com/liamghennigan/HyperFurion-VK
cd HyperFurion-VK
./packaging/macos/install-macos.sh

Native Quartz injection (full Unicode) and event-tap hotkeys, running as a launchd agent. macOS will ask for two permissions — Accessibility and Microphone — the installer walks you through both. Status arrives as notification-center toasts.

Windows — beta

git clone https://github.com/liamghennigan/HyperFurion-VK
cd HyperFurion-VK
powershell -ExecutionPolicy Bypass -File packaging\windows\install-windows.ps1

Native SendInput injection (full Unicode) and a low-level keyboard hook, started at login from your Startup folder — no admin required. Beta on both platforms means exactly that: unit-tested, honest about being young.

Linux needs the uinput module, Python 3.11+, and systemd user services; the installer supports apt-get, dnf, and pacman. Full walkthroughs, non-interactive install, and troubleshooting live in the README.
