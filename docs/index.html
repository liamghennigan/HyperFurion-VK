the claim

Transcription types words. A keyboard does everything else.

Every voice tool you've tried produces text and stops. A keyboard corrects, selects, formats, navigates, and commands — so a voice keyboard has to do the same, system-wide, in the app you're already in.

not this

A dictation box

A microphone pointed at a text area. You speak, it types, you grab the keyboard to clean up the mess.

not this

A chat window

An assistant that answers in its own app and leaves you to copy-paste the result to where you actually work.

this

An input device

A local daemon with a virtual keyboard. Spoken intent becomes keystrokes — typed, repaired, rewritten, or drafted wherever your cursor already is.

Ordinary dictation

–Converts speech into text, then stops.
–Works best inside its own dedicated text box.
–Leaves every mistake for your hands to repair.
–Treats corrections, commands, and rewrites as edge cases.
–Feels like a microphone feature.

how dictation works — molten — [flow]

Text is worked like metal.

With a streaming provider, words land in the focused field while you speak. Each word stays molten for a stability window — 1.5 seconds by default. If the provider revises it, the daemon backspaces to the divergence point and retypes. Once a word survives the window it freezes, and is never touched again.

t < 1.5 s State one

Molten

Just typed, still glowing. The provider may still change its mind about these words, and the daemon is allowed to reshape them.

backspace → retype State two

Repaired

A revision arrives: backspace to the divergence point, retype the corrected tail. The text heals itself in front of you, and repairs stay short by construction.

t ≥ 1.5 s State three

Frozen

Survived the window. Frozen words are permanent — your caret never runs away, and no late revision can rewrite what already set.

Context registers — [registers] the daemon probes the focused app at recording start (AT-SPI / Quartz / Win32) and picks how speech is rendered
proseSmart capitalization and punctuation spacing. The default.
terminalNo auto-caps, spoken numbers as digits, pastes with Ctrl+Shift+V. Known terminals map here automatically.
verbatimGrammar off; words exactly as recognized. Password fields force this — always.
pythonCompiles speech, no model in the loop."for i in range ten colon" → for i in range(10):
shellSame, for the pipe-and-flag dialect."pipe grep dash i error" → | grep -i error
the spoken grammar — [flow.commands]

Edits are commands, not text.

Corrections, punctuation, and escapes are parsed deterministically, before anything is typed — on every provider, streaming or not. The same phrase does the same thing every time, because the grammar runs in the daemon, not in a model.

You sayYou get
“scratch that” · “delete that”deletes the last utterance — even after it’s been typed
“new line” · “new paragraph”a line break · a blank line
“period” · “comma” · “em dash” · “open quote”the glyph, correctly spaced — thirty-odd phrases
“literal period”the word “period” — the escape hatch
“twenty three” · “three point one four”23 · 3.14 — in the terminal register, or with numbers = "always"

the engine room — [providers]

Any brain. Local hands.

A local daemon owns the hotkeys, the microphone, and a virtual keyboard — uinput on Linux, Quartz on macOS, SendInput on Windows. Speech recognition is a config value, not a lock-in.

Pick a provider

xAI is the default and what VK is daily-driven on. Swap providers or models in config — no code changes.

xai — default openai groq deepgram assemblyai elevenlabs (tts) hyperfurion (subscription)

The hyperfurion provider is one hfk_ key instead of provider accounts — same streaming protocol, relay fully self-hostable.

Or go fully offline

Point the OpenAI-compatible provider at any local server — Whisper, Parakeet, Voxtral, Kokoro. No key, no network, nothing leaves the machine.

# ~/.config/voice-keyboard/config.toml
[providers.openai]
base_url = "http://localhost:8000/v1"

[stt]
provider = "openai"   # local Whisper

[tts]
provider = "openai"   # local Kokoro

the covenant — enforced in code, not in copy

A keyboard this powerful has to know its place.

You own the Enter key.

An input device that can type anywhere must never act on its own. Every guarantee on the right is enforced mechanically — in the injector, the probe, and the defaults — not promised in marketing copy. Every claim on this page is lifted straight from the README the code ships with.

The machine may draft. Only a hand may send.

⏎ held — yoursvoice → drafted keystrokes → you press Enter
⏎
It never presses Enter for youCommand drafting types one line and stops. The refusal is enforced inside the keystroke injector on every path — keycode, newline, clipboard paste. Your keypress is the consent.
◉
Push-to-talk, by defaultNothing is captured until you summon it. The wake word is a tiny local detector, opt-in and off by default — and the hotkey remains the hard mute.
▣
Password fields are sacredThe focus probe sees the widget: a password field always forces verbatim, is never written to history, and never biases recognition.
⌖
It never types into the wrong windowIf focus moves mid-dictation, typing freezes instantly and the transcript lands on the clipboard instead.
◌
Audio lives in memoryRecorded speech is held in memory while it is transcribed and never written to disk. The optional history ledger is a local file, mode 600, off by default.
known limits — stated, not hidden

Honest limitations.

A keyboard you trust with every window has to be honest about its edges. These are the current ones.

Linux first. Python 3.11+, systemd user services, uinput access. macOS and Windows are beta — native Quartz / SendInput backends that say so plainly.
Cloud by default. No speech model ships in the box. Fully offline is one config line: point providers.openai.base_url at a local server.
Live typing needs streaming. REST providers land the text on stop — the grammar still applies, the words just arrive together.
Repairs assume a still field. Type or click into the text mid-dictation and a repair can land wrong. The stability window and max_molten_chars bound the damage.
“Scratch that” respects Unicode. It declines to delete across emoji and combining marks, where backspace-per-character isn’t reliable.
Frozen means frozen. A late revision that disagrees with an already-frozen word loses; only the molten tail adopts changes.
Non-ASCII pastes on Linux. Accents, CJK, and emoji go through the clipboard (wl-copy / xclip); clipboard managers may glimpse the transient entry.
The overlay is GNOME’s. The recording pill with live molten captions is GNOME Shell on Wayland; other desktops get ordinary notifications.
This page runs on your browser’s engines. Dictation uses the browser’s speech engine the way the daemon uses its provider; no engine or no mic, and the page says so and improvises — honestly labeled. Type real in the proof terminal to opt into the actual xAI engine, rate-limited and labeled while it runs.

the live config — ~/.config/voice-keyboard/config.toml

Edit it. The page re-binds.

On the desktop, [flow], [registers], and [llm] hot-reload at the next recording. This copy is live: slow stability_ms to 4000 and watch words stay molten longer, set auto_stop_ms = 1200 and just stop talking, rename the wake_word — then dictate again up in the proof.

[hotkey]
key = "control+alt+v"
# auto = tap to toggle, hold to talk
mode = "auto"
hold_threshold_ms = 280

[flow]
# false = the old behavior: record, stop, then type
live = true
# a molten word freezes after surviving this long
stability_ms = 1500
# 1200 = hands-free: tap, speak, done (0 = off)
auto_stop_ms = 0
# spoken cardinals → digits: auto | always | off
numbers = "auto"
# "<wake_word>, <instruction>" rewrites via [llm]
wake_word = "vk"

[registers]
# prose | terminal | verbatim — the daemon probes the
# focused app; the proof's terminal is always "terminal"
default = "prose"

[stt]
language = "en"
interim_results = true

[tts]
voice_id = "eve"
rate = 1.0
pitch = 1.0
